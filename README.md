# HackSwipe — Оценка хакатона в один клик + Swipe

**Решение для хакатона TulaHack 2026**

**HackSwipe** — это современная веб-платформа, которая полностью автоматизирует процесс оценки проектов на хакатоне. 

Жюри и организаторы могут быстро регистрировать критерии, ставить баллы, а система автоматически считает взвешенные итоги, сортирует команды и определяет победителей. Для удобства оценки добавлена **swipe-механика** (вправо/влево), как в Tinder, а также **реaltime-чат** для общения между участниками и жюри.

Проект сочетает в себе кейс **Задача 2** (HackRank) и элементы **Задачи 12** (Pet-swipe приложение).

## ✨ Основные возможности

- Ролевая модель: администратор, эксперты (жюри), участники/команды
- Регистрация и управление командами
- Конструктор критериев оценки с весами и максимальными баллами
- Удобная панель жюри с формой оценки
- Автоматический расчёт итоговых баллов и определение победителей
- Swipe-интерфейс для быстрой предварительной оценки
- Realtime-чат (Socket.IO)
- Публичная страница рейтинга с анимациями
- Поддержка медиафайлов (фото проектов)

## 🛠 Технологический стек

**Backend:**
- FastAPI (Python)
- Socket.IO (python-socketio)
- PostgreSQL + Alembic (миграции)
- AsyncPG

**Frontend:**
- Vue 3 + Vite
- Pinia (state management)
- Tailwind CSS + DaisyUI

**Деплой и инфраструктура:**
- Docker + Docker Compose
- Nginx (reverse proxy + раздача статики)
- Prometheus + Grafana + Loki + Promtail (мониторинг и логи)
- Telegram-бот (опционально)

## 🚀 Быстрый запуск (локально)

### 1. Клонируйте репозиторий

```bash
git clone https://github.com/yourusername/HackSwipe.git
cd HackSwipe
docker compose up --build
```

**База данных**
 - Миграции и БД развернута на сервере отдельно от приложениеи общается по HTTP.active
 - При желании, вы можете самостояльно развернуть БД, её сервис уже хранится в закомментированном виде в compose.yml 
 - Миграции мы делали при помощи Alembic. Вы же можете выбрать любой удобный вам инструмент. Все последние моедли для БД хранятся в папке /backend/app/models.active
 - Папка /backend/alembic не является нужной, в ней вы можеет просмотреть пример миграции и старые недействиетльные миграции.active

**Окружение и переменные окружения**
 - Настоятельно рекомендуем перед началом работы с пректом установить виртуальное окружение следубщими командами:

 - Для Windows:

```bash
python.exe -m venv .venv
```
 
 - Для MacOS или Linux:

```bash
python3 -m venv .venv
```

 - Также рекомендуем установить все зависимости, которые использовалиь в проекте. Мы использовали пакетный менеджер pip, поэтому и пример будем на нем. Все зависимотсти находятся в папке /backend/requirements.txt. Пример установки:

```bash
python.exe -m pip install --upgrade pip
cd backend 
pip install -r requirements.txt
```

 - Дальше можете спокойно работать с проектом, проблемы не возникнут.

 - Все важные зависимости вынесены в файлы окружения, такие как .env и в репозитории они не хранятся. Пример таких файлов с хранящимися там параметрами вы можете увидеть в папках /backend и /frontend и называются они .env.example. Таким же образом можете создать у себя на устройстве .env-файлы, откуда в ваш проект будут подтягиваться параметры. 

 - Все параметры, используемые в проекте, можете увидеть в файлике congif.py в папке /beckend/app/core. 

**Деплой**
 - Файлик compose.yml находится в корневом каталдоге и содержит все необходимые для раюоты приложения сервисы, среди них активными являются сервисы для backend, frontend и метрики(grafana, prometheus, grafana loki, promtail) - их конфиги находятся целиком и полностью в папке /infra, ручка(эндпойнт) для работы с ним в проекте имеет путь /metrics. Доступ к метрикам осуществляется по URL /opc/grafana. 

 - Так как переменных окружения нет в репоизтории - есть 2 пути решения:
 - 1 путь предполагает создание .env файлик в корневом каталоге, папках /backend и /frontend. Данный путь самый легкий, но небезопасный, так как при получении доступа к серверу, человек сможет спокойно их посмотреть.
 
 - 2 путь предполагает использование встроенной в Docker функции, которая называется Docker Secrets. С ее помощью вы можете вшить необходимые данные напрямую в контейнер. Таким образом, вы значительно увеличиваете безопасность вашего приложения, но немного усложняете процесс. Выглядеть это будет следующим образом:

```compose.yml
services:
 ....

secrets:
  internal_password:
    external: true
  password:
    external: true
  ..... 
```

 - Как-то так. Я больше склоняюсь ко 2 способу, так как возможность украсть данные значительно ументшается при таком способе и станет возможнна толтко при получении прав sudo на сервере.
